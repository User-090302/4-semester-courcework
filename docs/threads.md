# Назначение модуля

Модуль `threadWork.py` отвечает за управление многопоточностью в системе мониторинга ресурсов. Его основная задача — запуск и контроль фоновых потоков, в которых выполняются collectors (модули сбора системной информации).

Фактически данный модуль реализует упрощённый менеджер потоков, обеспечивающий параллельное выполнение задач без блокировки основного потока веб-сервера.

# Роль в архитектуре системы

В общей архитектуре проекта threadWork выполняет роль слоя исполнения (execution layer), обеспечивая:

 - запуск collectors в отдельных потоках;
 - асинхронное обновление данных;
 - изоляцию сбора данных от веб-логики.

Модуль используется ядром системы (`core.py`) для инициализации всех активных collectors.

# Основные компоненты
1. Класс threadWork

Основной класс модуля, реализующий управление потоками.

2. Атрибуты класса
`THRs` — список объектов потоков (`threading.Thread`), соответствующих каждому collector’у.
3. Методы класса
`__init__(self)`
Инициализирует пустой список потоков.

## `initTHR(self, thrs=[])`

Принимает список объектов collectors и создаёт для каждого поток, привязывая к нему метод run() соответствующего collector’а.

### Логика:

 1. проходит по списку collectors;
 2. создаёт threading.Thread для каждого;
 3. добавляет поток в список THRs.
## `StartTHRs(self)`

Запускает все предварительно инициализированные потоки.

### Особенности реализации:

1. выполняет проверку на `None`-значения;
2. пропускает неинициализированные элементы;
3. запускает потоки в daemon-режиме (завершаются вместе с основным процессом);
4. возвращает True при успешном запуске, False при исключении.


## `startTHR(self, index)`

Запускает отдельный поток по индексу в списке `THRs`.

Используется для точечного управления отдельными `collectors` (в рамках отладки или расширения функциональности).

# Взаимодействие с другими модулями

`core.py` — передаёт список collectors для запуска потоков
`collectors.py` — предоставляет метод run(), который выполняется в отдельном потоке
`web.py` — косвенно получает обновлённые данные, собранные потоками

# Особенности реализации

- Используется стандартная библиотека threading
- Потоки создаются в daemon-режиме (фоновые процессы)
- Архитектура ориентирована на простую параллелизацию без сложного планировщика задач
- Управление потоками реализовано на уровне списка без использования пула потоков или asyncio
# Ограничения

- Отсутствует механизм остановки потоков (stop / join)
- Нет контроля состояния потоков (alive / dead monitoring)
- Используется упрощённая модель управления без thread pool
- Нет защиты от повторного запуска потоков
- Отсутствует синхронизация доступа к общим данным (при расширении может возникнуть race condition)
