
### 1. Назначение модуля

Модуль `core.py` является центральной точкой запуска системы мониторинга ресурсов. Он отвечает за инициализацию конфигурации, создание компонентов системы (collectors, веб-сервер, менеджер потоков) и запуск основного цикла работы приложения.

Фактически модуль выполняет роль **ядра (orchestration layer)** всей системы.



### 2. Роль в архитектуре системы

`core.py` объединяет все подсистемы проекта:

- загрузка конфигурации (`config.jsonc`);
- инициализация collectors;
- запуск многопоточности через `threadWork`;
- запуск веб-сервера;
- связывание слоя данных и слоя представления.

Таким образом, данный модуль реализует центральный управляющий слой (application core).



### 3. Загрузка конфигурации

При старте системы выполняется чтение файла `config.jsonc`:

- `UPDATE` — частота обновления данных collectors;
- `HOST` — IP-адрес веб-сервера;
- `PORT` — порт веб-сервера;
- `MODE` — режим работы системы (html / api).

Конфигурация загружается один раз при запуске и используется всеми компонентами системы.



### 4. Класс WebServer (наследование)



#### 4.1. Назначение

Класс `WebServer` расширяет базовый класс `web.WebServer` и связывает веб-слой с системой collectors и потоков.



#### 4.2. Методы



##### `returnData(self, Collector)`

Возвращает данные collector’а, если он существует.

- выполняет проверку на `None`;
- вызывает метод `returnData()` у collector’а.



##### `service(self, arr=[])`

Отвечает за инициализацию потоковой модели:

- создаёт экземпляр `threadWork`;
- фильтрует список collectors (удаляет `None`);
- сохраняет активные collectors в `self.arr`;
- запускает потоки обновления данных.

Таким образом, метод связывает слой данных с системой многопоточности.



##### `_setup_routes(self)`

Определяет HTTP-маршруты Flask.

Основной маршрут `/`:

- формирует HTML-страницу;
- проходит по списку collectors;
- собирает их данные;
- подставляет результат в HTML-шаблон (`%items%`).



### 5. Класс core



#### 5.1. Назначение

Класс `core` является главным управляющим компонентом приложения. Он отвечает за полную инициализацию системы и выбор режима работы.



#### 5.2. Метод `__loadInfo()`

Выполняет создание всех collectors в зависимости от конфигурации:

- CPUCollector
- RamCollector
- NetCollector
- DisksCollector
- FansCollector

Если соответствующий модуль отключён в конфигурации — вместо него используется `None`.

Также формируется список `mainInfo`, содержащий все активные collectors.



#### 5.3. Логика выбора режима работы

В зависимости от параметра `MODE`:

- `html` — запускается веб-интерфейс на Flask;
- `api` — режим заявлен, но не реализован (заглушка).



#### 5.4. Метод `start()`

Запускает веб-сервер, вызывая метод `run()` у объекта `WebServer`.



### 6. Точка входа приложения

При запуске файла напрямую:

- создаётся экземпляр `core`;
- выполняется инициализация системы;
- запускается веб-сервер.



### 7. Архитектурная роль в системе

`core.py` выполняет функции:

- централизованного контроллера системы;
- связующего слоя между всеми модулями;
- точки запуска приложения;
- конфигурационного оркестратора.



### 8. Особенности реализации

- используется динамическая конфигурация через JSONC;
- collectors создаются условно (feature flags через config);
- используется многопоточность для обновления данных;
- веб-слой реализован через наследование базового сервера;
- система построена по упрощённой MVP-архитектуре.



### 9. Ограничения архитектуры

- отсутствует строгий интерфейс для collectors;
- используется упрощённая фильтрация `None`-значений;
- нет централизованного управления жизненным циклом потоков;
- режим API не реализован;
- присутствует смешение уровней ответственности (core управляет и данными, и запуском, и веб-слоем).
