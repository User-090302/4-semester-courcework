# Назначение модуля

Файл `config.jsonс` представляет собой централизованную конфигурацию системы мониторинга ресурсов. Он используется для задания параметров запуска приложения, включения/выключения модулей сбора системной информации, а также определения режимов работы и путей к ресурсам интерфейса.

Файл реализован в формате JSONC (JSON with comments), что позволяет добавлять поясняющие комментарии без нарушения структуры данных.

#  Роль в архитектуре системы

Конфигурационный файл является входной точкой настройки всей системы и используется модулем ядра (core) для:

определения активных collectors (модулей сбора данных),
выбора режима работы системы (HTML или API),
задания параметров сетевого сервера,
указания путей к HTML-шаблонам.

Таким образом, конфиг выполняет функцию параметризации всей системы без необходимости изменения исходного кода.

# Структура конфигурации
1. Блок enabled

## Определяет, какие подсистемы мониторинга активны:

`cpu` — мониторинг процессора
`ram` — мониторинг оперативной памяти
`net` — мониторинг сетевых интерфейсов
`fans` — мониторинг скорости вращения вентиляторов
`disks` — мониторинг дисковой подсистемы

Каждый параметр принимает логическое значение (true/false), что позволяет динамически включать или отключать соответствующие collectors.

# Блок setts

Содержит основные параметры работы системы:

`mode` — режим работы приложения:
    `html` — веб-интерфейс на основе HTML-рендеринга
    `api` — (в разработке) режим API-выдачи данных без UI
`update` — период обновления данных в секундах для collectors (тип float)
`logs` — флаг включения логирования (на текущем этапе не реализован)
`ip` — адрес, на котором запускается веб-сервер
`port` — порт веб-сервера
# Блок path

Определяет пути к ресурсам пользовательского интерфейса:

`htmD` — директория HTML-шаблонов системы
`mainP` — основной HTML-шаблон, используемый для формирования главной страницы
# Использование в системе

При запуске приложения файл конфигурации загружается модулем core.py и используется для:

- инициализации активных collectors,
- выбора режима запуска (HTML/API),
- настройки Flask-сервера,
- загрузки HTML-шаблонов интерфейса.

Изменение конфигурации не требует перекомпиляции или изменения кода, что позволяет гибко управлять поведением системы.

# Особенности реализации
- Используется формат JSONC, что позволяет добавлять комментарии (в стандартном JSON они недопустимы).
- Конфигурация централизована и загружается единожды при старте системы.
- Поддерживается модульное включение/выключение компонентов без изменения кода.
- Архитектура ориентирована на расширение (добавление новых collectors и режимов работы).
# Ограничения
1. Отсутствует валидация конфигурации при загрузке (ошибки структуры могут привести к сбоям).
2. Параметры logs и api находятся в стадии разработки и не используются полностью.
3. Нет поддержки горячего обновления конфигурации (перезапуск требуется для применения изменений).
