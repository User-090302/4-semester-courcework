
## Общая документация проекта (система мониторинга ресурсов)

---

### 1. Назначение проекта

Проект представляет собой кроссплатформенную систему мониторинга ресурсов операционной системы с веб-интерфейсом. Основная цель — предоставление информации о состоянии системы (CPU, RAM, сеть, диски, вентиляторы) через локальный HTTP-сервер.

Система ориентирована на локальное использование в сетевой среде и заменяет классические утилиты мониторинга (top, htop, task manager) веб-интерфейсом.

---

### 2. Общая архитектура

Архитектура системы построена по модульному принципу и включает следующие слои:

**1. Data Layer (слой сбора данных)**
- collectors.py
- Отвечает за получение системных метрик через psutil
- Работает в отдельных потоках

**2. Execution Layer (слой выполнения)**
- threadWork.py
- Управляет потоками collectors
- Обеспечивает параллельное обновление данных

**3. Application Core (ядро системы)**
- core.py
- Инициализация системы
- Связывание всех компонентов
- Управление режимами работы

**4. Presentation Layer (веб-слой)**
- web.py
- Flask сервер
- Генерация HTML интерфейса

---

### 3. Принцип работы системы

1. При запуске загружается конфигурационный файл `config.jsonc`
2. Инициализируются выбранные collectors
3. Каждый collector запускается в отдельном потоке
4. Collectors периодически обновляют данные системы
5. Flask-сервер отдаёт HTML-страницу
6. HTML-шаблон заполняется актуальными данными collectors

---

### 4. Источник данных

Система использует библиотеку `psutil` для получения информации:

- загрузка CPU по ядрам
- использование RAM и swap
- сетевой трафик по интерфейсам
- состояние дисков
- скорость вентиляторов (при наличии поддержки ОС)

Дополнительно используется `subprocess` для получения информации о CPU (в зависимости от платформы).

---

### 5. Интерфейс системы

Интерфейс реализован как HTML-шаблоны:

- данные вставляются через строковую замену плейсхолдеров;
- используется минимальный шаблонизатор (без Jinja2);
- поддерживается динамическая генерация блоков данных;
- итоговая страница формируется на стороне сервера.

---

### 6. Многопоточность

Система использует модель:

- один поток на каждый collector;
- daemon-потоки;
- периодическое обновление по таймеру;
- отсутствие централизованного планировщика задач.

Многопоточность применяется для асинхронного сбора данных без блокировки веб-сервера.

---

### 7. Конфигурация системы

Конфигурация задаётся через `config.jsonc`:

- включение/выключение collectors;
- выбор режима работы (HTML / API);
- настройка IP и порта;
- настройка интервала обновления данных;
- пути к HTML-шаблонам.

---

### 8. Режимы работы

**HTML mode**
- основной режим
- запуск Flask сервера
- отображение данных через веб-интерфейс

**API mode**
- заявлен, но не реализован
- предполагается выдача данных в JSON формате

---

### 9. Особенности реализации

- модульная архитектура (разделение на collectors, core, web, threading)
- использование psutil как универсального источника метрик
- генерация HTML на стороне backend
- простая модель потоков без thread pool
- конфигурационное управление функциональностью
- ориентация на MVP-реализацию

---

### 10. Ограничения системы

- отсутствует синхронизация доступа к общим данным (возможны race conditions)
- нет REST API реализации
- отсутствует безопасность доступа (нет авторизации)
- смешение уровней логики (данные + HTML в collectors)
- отсутствует масштабируемость на уровне распределённых систем
- нет hot-reload конфигурации
- зависимость от psutil и аппаратных возможностей ОС

---

### 11. Возможные направления развития

- выделение API слоя (REST/JSON)
- переход на шаблонизатор (Jinja2)
- замена threading на asyncio или task queue
- введение системы авторизации
- разделение data layer и presentation layer
- добавление WebSocket обновлений в реальном времени
- масштабирование на несколько хостов (distributed monitoring)

---

### 12. Итог

Проект представляет собой рабочий MVP системы мониторинга ресурсов с веб-интерфейсом. Архитектура упрощена, но модульно разделена и допускает дальнейшее расширение без полного переписывания системы.

Основная ценность проекта заключается в:
- практической реализации системного мониторинга;
- демонстрации работы с многопоточностью;
- интеграции backend + OS-level данных + web-интерфейса;
- возможности дальнейшей эволюции в полноценную распределённую систему мониторинга.
